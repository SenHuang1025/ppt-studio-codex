# Codex 协作方式

## 推荐工作方式
这套文档体系的目标，是让 Codex 在不同开发阶段只读取当前需要的上下文，避免一次吞入过长的大计划而失焦。

## 第一步：创建项目和文档
1. 创建项目仓库
2. 创建 `AGENTS.md` 与 `docs/` 全部文档
3. 提交一个初始文档基线

## 第二步：按 Phase 逐步推进

### 开启新 Phase
告诉 Codex：

```text
请阅读 docs/development-phases/phase-{n}-xxx.md，
我们现在开始这个阶段的开发。
请从任务 {n}.1 开始。
```

### 逐个任务推进
每完成一个小节并验证通过后，继续：

```text
任务 {n}.{m} 已验证通过。请继续任务 {n}.{m+1}。
```

### 遇到问题时

```text
任务 {n}.{m} 在 xxx 处遇到问题：{描述}。请排查并修复。
```

### 完成一个 Phase 后

```text
请更新 AGENTS.md 中的“当前开发阶段”为 Phase {n+1}。
```

然后提交一次阶段性变更。

## 第三步：Phase 之间的衔接
每个 Phase 完成后，让 Codex 做一次 review：

```text
Phase {n} 已全部完成。请检查代码质量，
确认没有遗漏的任务，然后我们开始 Phase {n+1}。
```

## 关键原则
1. **分步推进**：不要一次让 Codex 做太多事，一个任务节一个任务节推进。
2. **及时验证**：每完成一个任务后就验证，问题不要积压到后面。
3. **文档驱动**：`AGENTS.md` 是项目入口，编号文档是详细设计依据。
4. **上下文提醒**：如果 Codex 偏离方向，直接指向对应设计文档，例如 `docs/03-agent-system.md`。
5. **保持聚焦**：每次对话只聚焦当前 Phase 的当前任务。

## 这套方式的价值
- Codex 能按需阅读，不需要一次理解所有内容
- `AGENTS.md` 可以作为仓库级项目指令入口
- 分阶段开发更容易验收、回滚和补缺
- 遇到问题时可以回指具体文档章节
- 文档和代码能一起持续演进
