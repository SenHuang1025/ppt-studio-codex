# 页面设计与交互

## 路由结构

```text
/                          → Dashboard 项目列表
/project/:id               → 重定向到对应模式
/project/:id/chat          → Workspace 对话模式（创建 / 规划阶段）
/project/:id/preview       → Workspace 预览模式（浏览 / 优化阶段）
/settings                  → 设置页（独立路由，视觉上采用中心浮窗式工作台）
```

## 整体视觉方向

### 视觉关键词
- 暖色自然
- 纸感层次
- 轻磨砂架构
- 空间分层
- 专业创作工具
- 内容优先

### 设计结论
PPT Studio 的应用层 UI 采用“暖色、自然、纸感、轻质感工作台”方向：底层为暖纸渐层背景，中层为克制的轻磨砂容器，上层为高对比度内容工作区。

### 核心原则
1. 应用外壳可以有暖色渐层、轻磨砂、细腻阴影和轻微动态，但内容阅读区必须稳。
2. 大纲编辑区、设置表单主体、PPT 预览画布内部始终使用高对比度实色背景，不让背景光效污染内容。
3. 主交互色以松绿色系为主，强调和 CTA 使用琥珀色点亮，避免霓虹、赛博和深色玻璃观感。
4. 所有磨砂效果、高亮描边、阴影和圆角必须收敛为 design tokens，不允许组件各写一套。
5. 磨砂层级控制在 3 层以内，避免 Electron 桌面端出现滤镜性能问题。

## 信息架构与导航设计

### 应用层级

```text
底层：Ambient Background
    ↓
中层：左侧悬浮导航坞 + 页面级轻磨砂面板
    ↓
顶层：高对比度工作内容区（大纲编辑器、PPT 画布、设置表单）
```

### 全局导航
- 左侧使用悬浮式暖色导航坞 `AppDock`
- 导航坞固定包含：Dashboard、Settings、当前项目入口
- 当前项目进入 Workspace 后，不再通过传统顶部 tab 导航，而是通过顶部中央模式切换胶囊在“规划大纲 / 预览与调整”之间切换

### Workspace 模式切换
- 顶部中央使用滑动胶囊控制器 `ModeSwitchPill`
- 选中态是滑动式高亮底板，而不是简单文字变色
- 切换时使用轻微缩放和淡入淡出，体现空间切层，不使用突兀的左右整屏切换

## 页面级详细设计

### 3.1 Dashboard

#### 页面目标
- 建立产品第一印象
- 快速引导创建新项目
- 让最近项目可以被迅速继续编辑

#### 布局结构
- 左侧：悬浮导航坞
- 中央主区域：欢迎 Hero + 最近项目网格

```text
┌─ Dock ─┐  ┌─────────────────────────────────────────────────────────────┐
│ Home   │  │                    渐变标题 / 欢迎语                        │
│ Work   │  │         [ 大型输入 / 拖拽上传 / 新建项目联合入口 ]           │
│ Set    │  │                                                             │
└────────┘  │ 最近项目                                                     │
            │ ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐        │
            │ │ Project  │ │ Project  │ │ Project  │ │ Create   │        │
            │ └──────────┘ └──────────┘ └──────────┘ └──────────┘        │
            └─────────────────────────────────────────────────────────────┘
```

#### 主要区域说明
- 欢迎区：
  使用大字号渐变标题，文案聚焦“开始创作”
- 主操作区：
  合并“输入想做的 PPT 主题”“上传资料”“新建项目”三种入口
- 项目网格：
  使用轻纸感项目卡片展示最近项目，保留首屏轻盈感

#### 核心交互
- 点击 Hero 输入区可直接创建新项目并进入对话模式
- 支持拖拽文件到 Hero 区域，系统自动创建项目并进入工作区
- 点击项目卡片进入上次编辑状态
- 卡片菜单支持重命名、删除、归档、复制

#### 关键状态
- 空状态：
  使用 Hero Empty State，显示一个高质感引导插画 / 纸片层叠图形
- 加载中：
  项目网格使用轻纸感骨架卡片
- 错误：
  在 Hero 下方显示轻量错误条，不破坏版面
- 成功：
  创建项目成功后，卡片高亮并转场进入 Workspace

#### 视觉重点
- Dashboard 不做传统后台式页头表格布局
- 视觉重心放在中央 Hero，而不是左上角标题

### 3.2 Workspace - 对话模式

#### 页面目标
- 让用户感觉自己正在与 AI 共同规划一个演示文稿
- 左侧负责协作，右侧负责产出

#### 布局结构
- 左侧：轻磨砂 AI 对话区
- 右侧：高对比度大纲编辑区
- 顶部中央：模式切换胶囊

```text
┌─ Dock ─┐  ┌──────────────────── AI 对话区（轻磨砂） ────────────────────┐  ┌──────── 大纲编辑器（实色） ────────┐
│        │  │ AI 消息流 / 文件分析结果 / Thinking / 输入框 / 上传区          │  │ 项目标题  页数  主题建议            │
│        │  │                                                               │  │ Page 1                            │
│        │  │ [ 清晰描边输入框 ] [ 上传 ]                                    │  │ Page 2                            │
│        │  └───────────────────────────────────────────────────────────────┘  │ Page 3                            │
└────────┘                                                                      │ [ 确认大纲 ] [ 继续调整 ]          │
                                                                                └──────────────────────────────────┘
```

#### 主要区域说明
- 对话区：
  放置流式消息、文件卡片、分析摘要、系统状态、输入框
- 大纲区：
  放置项目名称、页数、主题建议、页面条目列表和确认操作

#### 核心交互
- 拖拽或点击上传文件
- 输入需求后，Agent 先分析文件再生成大纲
- 如果开启多智能体思辨模式，对话区需要能展示“草案 / 评审 / 综合”这类阶段性状态
- 右侧大纲可展开详情
- 用户点击“确认大纲”后进入预览模式

#### 关键状态
- 空状态：
  左侧显示对话欢迎语；右侧显示“大纲尚未生成”的引导面板
- 分析中：
  对话底部出现细线进度条，文件卡片显示解析状态
- 大纲已生成：
  右侧大纲编辑器显示确认 / 调整操作
- 错误：
  错误消息放在对话流里，同时右侧保留上一个有效大纲

#### 视觉重点
- 左侧是暖色轻磨砂材质，右侧是低透明或纯实色工作面
- 通过材质差异明确“讨论区”和“结果区”

### 3.3 Workspace - 预览模式

#### 页面目标
- 聚焦当前页的生成质量
- 支持“看一页、改一页、确认一页”的工作方式

#### 布局结构
- 左侧：缩略图轨道
- 中央：PPT 画布预览区
- 右侧：当前页操作和优化对话

```text
┌─ Dock ─┐  ┌──── 缩略图轨道（轻磨砂） ───┐  ┌──────────── PPT 预览核心区 ────────────┐  ┌──── 单页优化侧栏（轻磨砂） ───┐
│        │  │ [1] [2] [3*] [4] ...       │  │ 项目标题 / 页码 / 全屏 / 导出          │  │ 当前页：第 3 页               │
│        │  │ 状态色 + 选中描边           │  │ ┌────────────────────────────────────┐ │  │ 快捷操作                      │
│        │  │                             │  │ │            16:9 iframe            │ │  │ 对话历史                      │
│        │  │                             │  │ └────────────────────────────────────┘ │  │ 输入框                        │
└────────┘  └─────────────────────────────┘  │      [ 上一页 ] 3 / 12 [ 下一页 ]      │  └──────────────────────────────┘
                                             └─────────────────────────────────────────┘
```

#### 主要区域说明
- 缩略图轨道：
  负责页码导航、状态感知、拖拽排序、右键菜单
- 预览核心区：
  负责展示当前页，视觉中心必须绝对稳定
- 单页优化侧栏：
  顶部放快捷操作，中部放当前页状态与最近修改，底部放对话输入

#### 核心交互
- 点击缩略图切换页面
- 键盘左右键翻页
- 右侧发送修改意见时自动绑定当前页码
- 快捷操作可直接触发换配色 / 换布局 / 换动画 / 重新生成 / 确认本页
- 页面修改成功后，当前页边框出现一次性描边反馈

#### 关键状态
- 生成中：
  缩略图轨道按页显示“待生成 / 生成中 / 已生成”
- 当前页处理中：
  右侧侧栏标题明确显示“正在优化第 N 页”
- 页面已更新：
  iframe 外边框出现一圈暖色渐变扫线
- 无页面：
  中央显示生成引导，不展示空白 iframe

#### 视觉重点
- 中间预览区不是半透明面板，而是一个实心内容屏幕
- 预览容器阴影应明显弱于对话区外壳阴影，避免主次颠倒

### 3.4 Settings

#### 页面目标
- 让用户快速完成模型、主题、存储位置等系统设置
- 保持高级感，但不牺牲表单可读性

#### 布局结构
- 路由层面仍是独立页面
- 视觉层面为居中的大型暖纸设置窗口，背后保留 Ambient Background

```text
┌─────────────────────────────────────────────────────────────┐
│ 设置工作台                                                  │
│ ┌──────── 侧边分组导航 ────────┐ ┌──────── 表单主体 ───────┐ │
│ │ 模型设置                      │ │ Provider / Model        │ │
│ │ 推理模式                      │ │ 多智能体思辨开关         │ │
│ │ 默认主题                      │ │ API Key                 │ │
│ │ 存储路径                      │ │ API Base URL            │ │
│ │ 关于                          │ │ 测试连接 / 保存         │ │
│ └──────────────────────────────┘ └─────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

#### 主要区域说明
- 左侧分组导航：
  负责一级设置分类切换
- 右侧表单：
  负责所有输入控件和保存行为

#### 核心交互
- 切换 Provider 时联动模型列表
- 提供“开启多智能体思辨模式”开关，并在旁边说明其会提高质量但增加延迟与成本
- 该开关作为 Phase 2 对现有 Settings 页的增量扩展接入，不要求回改已完成的 Phase 1 基础设置链路
- API Key 支持显示 / 隐藏和测试连接
- 存储路径支持选择本地目录

#### 关键状态
- 加载中：
  表单区域使用骨架条
- 保存中：
  顶部或底部显示轻量进度提示
- 错误：
  错误提示紧邻字段，不只弹 toast
- 成功：
  显示短暂成功反馈并保留当前输入

#### 视觉重点
- 外壳可轻磨砂，表单内层必须稳定实色
- 不使用过多高亮按钮，避免设置页像娱乐场景

## 核心组件设计

### AppDock
- 用途：全局主导航
- 结构：Logo、导航图标、当前项目入口、底部辅助操作
- 状态：常态、hover、激活、收起
- 视觉建议：暖色轻磨砂导航坞 + 细腻纸感边缘

### ModeSwitchPill
- 用途：Workspace 两种模式切换
- 结构：胶囊容器、滑动高亮层、两个模式标签
- 状态：选中、hover、切换中
- 交互建议：高亮底板平滑滑动，配合内容区轻微 fade

### ProjectCard
- 用途：项目展示入口
- 结构：缩略图、标题、状态、更新时间、菜单
- 状态：默认、hover、选中、处理中
- 视觉建议：轻纸感卡片，hover 时整体上浮 2px 并增强边缘对比

### StatusBadge
- 用途：显示规划中 / 生成中 / 编辑中 / 已完成
- 结构：胶囊标签、状态文字、可选小图标
- 状态：处理中、成功、警告、归档
- 视觉建议：处理中用主色浅底，完成态用低饱和绿 / 琥珀强调

### ChatMessage
- 用途：渲染用户、AI、系统消息
- 结构：头像、气泡、元数据、消息体
- 状态：流式输出、完成、错误
- 视觉建议：用户消息用主色实体气泡，AI 消息用浅色轻面板气泡

### FileUploadArea
- 用途：拖拽上传和点击上传
- 结构：图标、提示文案、支持格式说明
- 状态：默认、dragover、uploading、error
- 视觉建议：边框使用虚线 + 轻强调，不做厚重卡片

### FileCard
- 用途：展示已上传文件及解析状态
- 结构：文件名、类型图标、大小、状态、删除按钮
- 状态：待解析、解析中、已解析、失败
- 视觉建议：轻纸感卡片 + 状态点，不要大面积彩色底

### OutlinePanel
- 用途：展示和确认 PPT 大纲
- 结构：项目摘要、页面列表、底部操作区
- 状态：空状态、生成中、已生成、待确认
- 视觉建议：外壳可带轻磨砂边框，内部列表区域使用实色

### OutlinePageItem
- 用途：单页大纲信息
- 结构：页码、标题、类型、摘要、展开详情
- 状态：折叠、展开、当前聚焦、已确认
- 视觉建议：简洁分割线，不做过度装饰

### SlideRenderer
- 用途：渲染当前 PPT 页面
- 结构：预览容器、iframe、加载遮罩
- 状态：加载中、可见、失败、页面更新中
- 视觉建议：纯实色预览底板 + 明确阴影边界

### SlideControls
- 用途：翻页与页码控制
- 结构：上一页、页码、下一页、键盘提示
- 状态：可点、禁用
- 视觉建议：按钮使用低干扰浅色按钮，不抢预览视觉中心

### ThumbnailNav
- 用途：页码导航、重排、状态感知
- 结构：缩略图列表、状态标识、添加页入口
- 状态：选中、待生成、生成中、已确认
- 视觉建议：纵向轨道布局，选中卡片显示精准描边和浅色底

### PageOptimizeChat
- 用途：针对当前页的优化对话
- 结构：当前页标题、消息列表、输入框、发送按钮
- 状态：当前页处理中、成功、错误
- 视觉建议：侧栏轻磨砂外壳，消息内容区保证足够对比度

### QuickActions
- 用途：提供高频快捷优化动作
- 结构：按钮组、当前页状态、版本入口
- 状态：默认、hover、处理中、禁用
- 视觉建议：图标按钮采用浅色轻面板 + 细描边，不使用笨重色块

### VersionHistoryDrawer
- 用途：展示当前页版本历史
- 结构：版本列表、时间、修改描述、预览 / 回滚操作
- 状态：展开、关闭、回滚中
- 视觉建议：高层级轻磨砂抽屉，内部列表区域仍用实色内容面

### SettingsForm
- 用途：系统设置录入
- 结构：字段区、帮助说明、测试连接、保存操作
- 状态：编辑、校验错误、保存中、成功
- 视觉建议：遵循 Naive UI 组件样式，但统一接入项目 token


### PPT 页面 Token
- PPT 页面主题仍沿用 `--slide-*` 命名体系
- 应用壳体 Token 和 PPT 页面 Token 分离维护，避免外层壳体风格误伤页面内容

### 字体层级
- 标题：`Inter` 或 `Roboto Flex`
- 正文：`Inter`
- 代码 / 流式状态 / 页码辅助信息：`JetBrains Mono`

### 圆角、阴影、间距

```text
--radius-md: 12px
--radius-lg: 16px
--radius-xl: 20px
--radius-2xl: 24px

--shadow-glass-1: 0 10px 24px rgba(111, 87, 48, 0.10)
--shadow-glass-2: 0 18px 42px rgba(111, 87, 48, 0.14)
--shadow-canvas: 0 20px 52px rgba(111, 87, 48, 0.14)
```

### UnoCSS / CSS Variables 实现建议
- 将 `glass-panel`、`glass-panel-strong`、`highlight-ring-primary`、`gradient-text-primary` 等抽成 shortcuts
- 只允许通过 token 访问颜色和阴影，不直接写零散 `rgba`

## 动效与反馈设计

### 页面切换
- 页面进入：`translateY(8px) + opacity`
- 面板切换：`scale(0.98 -> 1)` + `opacity`
- 不使用大幅度滑屏动画

### AI 思考 / 生成反馈
- 使用细线扫描条替代传统 spinner
- 对话流式输出时，气泡尾部保留微弱呼吸光点
- 逐页生成时，缩略图从骨架态逐步显影为真实页面

### 页面更新反馈
- 当前页更新时，iframe 边缘出现一次性环形扫线
- 成功后右侧消息区给出“已更新第 N 页”的明确文本

### 交互态建议
- hover：微上浮 + 边框提亮
- active：轻微下压 + 降低阴影
- disabled：降低透明度，不再保留强调态
- selected：使用清晰描边和浅色底，不只靠背景色变化

## 关键场景专项设计

### 场景 1：首次进入应用，无任何项目
- 中央显示 Hero Empty State
- 包含标题、副标题、上传入口、新建项目主按钮
- 下方可补一行“支持 Excel / PDF / Word / 图片”等格式说明

### 场景 2：上传多个文件后，AI 正在分析
- 对话区展示逐文件解析状态
- 文件卡片单独显示解析进度和成功 / 失败状态
- 顶部或底部使用低干扰扫描式进度反馈

### 场景 3：AI 已生成大纲，等待用户确认
- 右侧大纲区高亮确认按钮
- 左侧对话区出现简要总结消息
- 模式切换胶囊提示用户下一步将进入预览生成

### 场景 4：AI 正在逐页生成 PPT 页面
- 左侧缩略图轨道显示逐页状态
- 中央预览区可即时查看已完成页面
- 右侧侧栏显示当前生成页信息

### 场景 5：用户在预览模式下针对第 3 页发送修改意见
- 右侧标题明确显示“当前优化：第 3 页”
- 第 3 页缩略图保持选中描边态
- 其他区域透明度轻微降低，视觉聚焦到当前页和右侧输入区

### 场景 6：某一页修改后用户如何感知成功
- iframe 边缘出现一次扫线
- 对话流中出现“已修改第 N 页”确认消息
- 缩略图从“处理中”切回“已生成 / 待确认”

### 场景 7：导出中状态反馈
- 使用专用导出进度弹层或右上角任务面板
- 展示导出页数、当前阶段、预计剩余时间
- 导出完成后给出明确下载入口

## 桌面端适配建议

### 常见分辨率建议
- `1440x900`：
  Dock 72-80px，缩略图轨道 180-200px，右侧栏 320-340px
- `1600x900`：
  Dock 80-88px，缩略图轨道 200-220px，右侧栏 340-360px
- `1920x1080`：
  Dock 88-96px，缩略图轨道 220-240px，右侧栏 360-400px

### 弹性与固定策略
- Dock 固定宽度
- 缩略图轨道固定宽度
- 右侧单页优化栏固定宽度
- 中央 PPT 预览区弹性伸缩，并始终保证 16:9 内容优先

### 最小窗口
- 最小建议宽度：`1280px`
- 低于 `1280px` 时，不保证三栏布局体验；如需支持，右侧优化区应折叠为抽屉

### 性能约束
- 全局背景必须使用纯 CSS / SVG 渐变，不使用视频背景
- `backdrop-filter` 叠层不超过 3 层
- 动画优先基于 `opacity` 与 `transform`

## 给前端开发的实现注意事项
- `AmbientBackground`、`GlassPanel`、`ModeSwitchPill` 优先做成基础组件，不要在页面里重复拼效果
- Naive UI 只负责基础交互和表单能力，应用视觉统一由 design tokens + 外层包装控制
- 对话区、缩略图轨道、设置浮窗可以使用轻磨砂材质；大纲编辑器和 PPT 画布内部不要使用磨砂滤镜
- 任何全局动效如果影响文本清晰度或造成掉帧，优先降级效果，而不是强保视觉

## 给产品 / 设计评审的设计亮点总结
1. 应用外壳具有自然、专业的创作工具气质，但内容工作区保持专业稳重。
2. 对话、大纲、预览三类信息通过材质和空间层级被清晰区分，用户更容易理解当前在做什么。
3. 等待 AI 生成的过程被设计为“有反馈的创作过程”，而不是单调的技术等待。
